## HIGH 指摘 (ローンチブロッカー)

### H-1. ステータスバナーと「ステータスページ」の実体が未定義
- 該当箇所: §12.4「障害告知」/ §11.1 通知レベル
- 問題:
  - 「ステータスバナー: アプリ最上部、閉じれない (critical のみ)」とあるが、**何を critical と判定するか**、**誰が** バナーを上げ下げするか、**ソース・オブ・トゥルース** (status.linkdinside.com 等) が一切書かれていない。
  - 「ステータスページへの導線」とだけ書かれており、URL / 配色 / インシデント開始/進行中/解消の 3 状態の UI が無い。
  - critical 中にユーザーが画面操作した場合、進行中ジョブの挙動 (送信を止めるのか) が未定義。
- 推奨:
  1. `status.{domain}` を SoT として定義し、JSON フィードを App 側が 30s ごとに pull → グローバル `<StatusBanner />` で表示。
  2. インシデントレベルを `none / degraded / partial-outage / major-outage` の 4 段階とし、major-outage 時は `送信系 API はクライアント側でも 503 として扱い受信箱は read-only` の挙動を仕様化。
  3. ステータスバナー内に常に `incident_id` (例: INC-2026-0512-001) と「最終更新」相対時刻、ステータスページ外部リンクを併記。
- 期待効果: 「アプリ落ちてる?」サポート問い合わせを 50% 以上削減。インシデント中の二次データ破損を抑止。

### H-2. incident_id の生成元・突合動線が UI にしか出てこない
- 該当箇所: §12.3「5xx は…インシデント番号 INC-XXXX を控えてサポートへ」
- 問題:
  - incident_id を **ユーザーに見せろ** とは書いてあるが、これがバックエンドの trace/correlation_id と一致するのか、Sentry の event_id なのか、自前の INC-Sequence なのかが不定。
  - サポート画面 (`🆘 ヘルプ / 改善要望`) から incident_id を貼り付けて検索する UI がない。`/audit` も WHO/WHEN ベースで incident 検索は無い。
- 推奨:
  1. `correlation_id` を全リクエストに必ず付与 (フロント生成 ULID、サーバ往復継承)。`incident_id` はサーバ側で 5xx/safe_mode 時にこの correlation_id を含む形で発番。
  2. すべてのトーストとエラーモーダルに incident_id を **コピーボタン付き** で表示。`navigator.clipboard.writeText` 失敗時のフォールバックも仕様化。
  3. ヘルプフォームに「incident_id があれば貼ってください」フィールドを必ず置き、サポート側は incident_id → ログ/Sentry 直リンクの内部ツールに繋ぐ。
- 期待効果: MTTR 短縮 (incident_id 1 つで Sentry / 構造化ログ / Webhook ログ / Unipile 履歴を横串)。

### H-3. Critical 通知の「強制 ON」が UI 上で外せてしまう懸念
- 該当箇所: §11.1 通知レベル「Critical … ON 強制」/ §17 セキュリティ UX
- 問題:
  - 表に「ON 強制」とあるが、**通知センター S17 でユーザーがトグルできるかどうか**の仕様が曖昧。「強制」を UI で表現する手段 (disabled toggle + tooltip) が記述されていない。
  - エスカレーション SLA が未定義 (例: 安全モード発動後 5 分応答なし → メール、15 分 → Slack、30 分 → SMS / PagerDuty)。
  - チャネル単位の冗長化 (メール BounceBack 時の代替) が無い。
- 推奨:
  1. Critical チャネル (in-app / email / Slack) は **disabled トグル + 「Owner により強制 ON されています」tooltip** とし、Owner 1 名以上にメール再到達確認を 24h ごとに実施。
  2. 安全モード発動・SSO 期限切れ・課金失敗・プラン上限到達は Critical とし、`Slack incoming-webhook` / `Email DLQ` の二重化を仕様に明記。
  3. Critical 未読が 30 分以上残ると 2 次エスカレ (別 Owner にも CC) を仕様化し、UI の通知センターに「エスカレ済み」バッジを出す。
- 期待効果: アカウント BAN や請求停止のような不可逆事故を未然に防止。

### H-4. プラン上限到達時の「従量課金で続行 ↔ 停止」切替に保護機構なし
- 該当箇所: §6.8 S14 プラン / 請求「停止 / 従量課金 切替可。既定は **停止**」
- 問題:
  - 既定が停止なのは正しい (Safe Defaults) が、**従量課金 ON への変更は破壊的経済操作**にも関わらず、Undo/2FA/承認フローの言及が無い。
  - 上限 80% / 90% / 100% の事前警告 UX、上限到達時の進行中ジョブの挙動 (中断 vs 完走) も未定義。
  - 「ジョブを止めずに従量課金で続行」を ON にしたまま放置 → 月末請求が想定の数倍、というクラシックな事故パスが開いている。
- 推奨:
  1. 従量課金 ON への切替は Owner 限定 + 二要素再認証 + 「月次予算上限 (¥)」入力を必須化。月次上限超過で自動 OFF に戻す。
  2. 80% / 95% / 100% の通知 (in-app + email + Slack) を Critical に分類。100% 到達時はキャンペーン送信ジョブを atomic に停止 (受信箱は read-only で残す)。
  3. 「従量課金中」はダッシュボード上部に持続バナー (赤ではなく注意色) で常時可視化、当月の従量超過 ¥ もリアルタイム表示。
  4. 上限到達 → 自動停止イベントは監査ログ + Sentry transaction として残し、incident_id 付きで Owner にメール。
- 期待効果: 請求事故とユーザーの信頼喪失を防止。返金対応工数を削減。

### H-5. ユーザーが見える「失敗ログ・再試行」の動線が CRM 失敗ログのみで断片的
- 該当箇所: §6.6 S12 CRM 連携「失敗ログ: 直近 24h で 3 件 [詳細]」/ §21.3 API 連携面
- 問題:
  - 失敗ログ UI は CRM のみに局所化されている。LinkedIn 送信失敗 / Unipile Webhook 取りこぼし / AI 生成失敗 / メッセージ配信失敗の **横断ログ画面が無い**。
  - §21.3 で「未届のリトライ UI を保持」とだけ書かれているが、何件溜まっているか / 自動リトライか手動リトライか / DLQ の概念が UI に存在しない。
  - 送信ボタンの 2 段階 Undo (3 秒) しか書かれていないが、**送信後の配信ステータス** (送信済 / 配信失敗 / Unipile 取込失敗) を会話画面で確認する UI が無い。
- 推奨:
  1. 共通の **「配信履歴 / 失敗ログ」画面** (`/operations/delivery-log`) を新設。フィルタは `account / channel / error_class / time`。各行に correlation_id・再試行ボタン・「サポートに送る」。
  2. 各メッセージバブルに `送信中 / 送信済 / 配信確認済 / 配信失敗 (理由)` の 4 状態を表示。失敗時はインライン再試行 + 自動バックオフ残時間を表示。
  3. Unipile webhook 欠損検知 (定期 reconcile) のサマリをダッシュ「注意が必要なもの」に統合。
- 期待効果: ユーザーの「送ったはずなのに届いてない」問い合わせ削減、サポート工数▲ 30–40%。

## MEDIUM 指摘

### M-1. 429 / Unipile 4xx 時の UX が「待機中」止まりで自助手段がない
- 該当箇所: §21.3「レート 429: 自動バックオフ、UI には『混み合っているため待機中』」
- 問題: 何分待つのか、いつ復帰見込みか、ユーザーが何もしなくていいのか / 何かすべきか、が不明。`Retry-After` ヘッダ値の UI 反映が未定義。
- 推奨: バックオフ残時間カウントダウン + 「先に他の操作に進む / 通知で知らせる」CTA。Unipile 4xx は エラー分類別 (auth / rate / regulatory) で文言と推奨アクションを切り替える。
- 期待効果: 「動かない不安」起因の手動連打 (=さらに 429) を抑制。

### M-2. LLM 障害時のフォールバックが「人間ドラフトテンプレ」のみで具体性に欠ける
- 該当箇所: §9.3「AI が止まったら『人間ドラフト』テンプレに即フォールバック」
- 問題: テンプレの選択基準・編集体験・モデル単一障害 (OpenAI のみ等) 時の代替プロバイダ動線が無い。「責任ある言葉で原因と回避策」と書かれているが、**原因分類のカタログ** が無いと文言が場当たりになる。
- 推奨:
  1. LLM フォールバック階層: Primary → Secondary (別ベンダ) → ルールベース文 → 「下書き不可」表示 + テンプレ提供。各段の閾値 (latency / error_rate) を仕様化。
  2. エラー分類辞書 (timeout / safety_filter / quota / unsupported_lang) と各文言を i18n リソースで管理。
- 期待効果: AI ベンダ片肺障害 (年に数回発生する) でも業務継続。

### M-3. SSE / WebSocket 切断時の整合性回復手順が口頭説明のみ
- 該当箇所: §11.2「切断時はバナー…自動再接続」
- 問題: 再接続後の「未取得イベントの再同期」プロトコル未定義。受信箱バッジが古くなったまま、reconcile されない可能性。Optimistic UI の rollback 動作も曖昧。
- 推奨: 再接続時に `?since=<last_event_id>` で差分取得 → 取得完了で「最新化されました」トースト。10 秒以内に再接続できない場合は read-only モードに切替えて誤送信を防ぐ。
- 期待効果: 「返信に気づかなかった」インシデントの抑止。

### M-4. SLO 文書がそもそも存在しない (SLI/SLO/エラーバジェット)
- 該当箇所: 全体 (§15 はパフォーマンス予算のみ)
- 問題: §15 は LCP/INP/CLS など UX パフォーマンス指標で、**サービス可用性 SLI/SLO** (送信成功率 / Webhook 反映 P95 / AI 生成成功率) が無い。SLO 違反 → エラーバジェット燃焼 → 機能凍結のガバナンスが UI 通知設計と紐付いていない。
- 推奨: `/admin/slo` 画面 (Admin 限定) を仕様化。主要 SLO: 送信成功率 99.0% / Webhook 取込 P95 60 秒 / AI 生成成功率 98% / 安全モード誤発動率 < 0.5%。違反時はダッシュ「注意が必要なもの」に Owner 向け表示。
- 期待効果: SLA 提示の根拠ができ、Enterprise 商談で詰まらない。

### M-5. 監査ログ画面に「インシデント / 自動アクション」の流入が無い
- 該当箇所: §6 / §17 監査ログ
- 問題: 監査ログは WHO/WHEN/WHAT/IP/DIFF となっており、**system actor (自動停止 / 自動 OFF / 自動エスカレ)** が表現されない可能性。インシデントレポートとの相互リンクが無い。
- 推奨: `actor_type = user | system | external_webhook` を必須カラムに。system 行には incident_id / cause を必ず関連付け、`/audit?incident_id=INC-...` で絞り込めるように。
- 期待効果: 監査・コンプライアンス対応の所要時間短縮。

### M-6. ウォームアップ計画の「自動停滞」中の通知ロジック未定義
- 該当箇所: §10.2「条件未達なら自動で停滞」
- 問題: 停滞 → 何日間停滞すると Warning 通知 → さらに何日で Owner 強制エスカレ、というしきい値設計が無い。停滞中のキャンペーンの扱い (一時停止 vs 縮小運転) も曖昧。
- 推奨: 停滞 3 日 → ユーザー (Operator) に Action Required、7 日 → Owner にエスカレ、14 日 → 自動でキャンペーン PAUSED + サポート起票。
- 期待効果: BAN 直前のサイレントフェイル防止。

### M-7. 「safe_mode.triggered」イベントの後段 (解除/誤検知率) の観測軸が PostHog に無い
- 該当箇所: §16.1
- 問題: 発動イベントだけで、解除 / 誤検知 / 再発がイベント化されていない。誤検知率を SLO に組み込めない。
- 推奨: `safe_mode.resolved (resolution_type, duration_min, was_false_positive)` を追加。`was_false_positive=true` が増えたら閾値を見直すフローを Runbook 化。
- 期待効果: 安全モード閾値チューニングが定量化できる。

## LOW 指摘

### L-1. ダッシュボードの「規約注意 1 件」の判定根拠が不可視
- 該当箇所: §6.1「規約注意 1 件」
- 推奨: ホバーで「Unipile から 4xx + パターン X」のような根拠と incident_id を提示。

### L-2. /audit のリテンションが「90日 / 13ヶ月」のみで法務要件と紐付かない
- 該当箇所: §6.10
- 推奨: 個人情報保護法 / GDPR の保管要件と照合した根拠コメントを UI 上の i アイコンで表示。

### L-3. 通知センターでの「過去のインシデント履歴」ビューが無い
- 該当箇所: §11.3
- 推奨: `/notifications?type=incident` でインシデント履歴一覧、各行から incident_id 付きの postmortem 公開ページへリンク。

### L-4. キーボードショートカット (`J/K/R/E/S`) の衝突確認・カスタマイズ未対応
- 該当箇所: §6.4 S10
- 推奨: ヘルプ `?` で全ショートカット表示、IME 中は無効化 (compositionend 後に有効化)。

### L-5. CSV エクスポート権限と incident_id の関係性が無い
- 該当箇所: §17 「危険操作: 二要素再認証」
- 推奨: CSV エクスポートは監査ログに `export_id` を発番、ダウンロード URL は短期署名 (15 分 TTL)。

### L-6. データ削除 30 日 grace の「キャンセル」UI 動線が記述薄い
- 該当箇所: §6.11 S20
- 推奨: 削除予約中はバナーで「あと N 日で物理削除」を全画面で常時表示し、Owner のみキャンセル可。

### L-7. パフォーマンス予算に「サーバ側 P95」が無い
- 該当箇所: §15
- 推奨: `/api/messages POST` P95 < 800ms など、UX 影響の大きい API のサーバ側 SLI を追加。

## 良い点 3 つ

1. **Safe Defaults を原則に格上げ** (§1.2) — 自動送信 OFF / 24h Undo / 7d 復元 が原則として明文化されており、SRE が好む「破壊操作はデフォルト不可」の文化が UI 仕様レベルで担保されている。
2. **状態機械を IA に揃えて単一語彙化** (§3.3 / §21.2) — `DISCOVERED→...→FAILED` を全画面同色・同アイコン・同ラベルで使う方針は、インシデント時の「どの状態で止まっているか」のオペレーター判断を高速化する。
3. **送信は Optimistic UI を禁止** (§11.2) — 「送信は Optimistic 禁止」の一文は、誤送信→消えないインシデント化の最大要因を仕様で封じている。SRE 観点では非常に評価できる。

## ローンチ前に必須の Runbook 5 項目

下記 5 つは **Phase 1 (MVP) ローンチまで** に必ず文書化・訓練 (game day) を実施する。

1. **RB-01 安全モード発動時オペレーション**
   - 入力: `account_id`, `incident_id`, `reason`
   - 手順: ① Unipile 状態確認 → ② 同テナント他アカウントへの波及確認 → ③ Owner エスカレ (Slack / メール) → ④ ユーザーへの説明テンプレ送付 → ⑤ クールダウン明け再開判定 → ⑥ postmortem 起票
   - SLA: 検知から Owner 通知まで 5 分、ユーザーへの初報まで 30 分。

2. **RB-02 Unipile 障害 / OAuth 期限切れ大量発生時**
   - トリガ: Unipile 4xx/5xx > 5%/5min または OAuth 期限切れ > 10 アカウント/h
   - 手順: ① ステータスバナーを degraded で点灯 → ② 送信ジョブを全テナント一時停止 (受信箱は read-only 維持) → ③ Unipile ステータスページ確認・サポート起票 → ④ 復旧後にバックフィル順序を決定 (古いウォームアップ中アカウントから低レートで再開)
   - SLA: degraded 切替を 2 分以内、ユーザー告知を 10 分以内。

3. **RB-03 LLM ベンダ障害時のフォールバック切替**
   - トリガ: Primary LLM error_rate > 10%/2min または P95 latency > 30s
   - 手順: ① Secondary プロバイダへ自動フェイルオーバー (機能フラグ) → ② AI Draft 案数を 3→1 に縮退 → ③ ユーザーには「現在 AI ドラフトの精度が低下しています」を会話画面に明示 → ④ Primary 復旧後 24h は段階的に戻す
   - SLA: フェイルオーバー切替まで 3 分以内。

4. **RB-04 プラン上限到達 / 従量課金暴走の止血**
   - トリガ: 従量超過 ¥ がテナント月次予算上限を超過 / または `safe_mode.triggered` と同時に課金イベント増
   - 手順: ① 従量課金フラグを atomic に OFF → ② 進行中ジョブを安全停止 → ③ Owner にメール + Slack で incident_id と当月課金見込みを通知 → ④ 翌営業日 Owner と支払い確認後に再開判定
   - SLA: 予算超過検知から従量 OFF まで 1 分以内 (自動)。

5. **RB-05 incident_id 発番 / postmortem 公開フロー**
   - トリガ: critical / major-outage / 安全モード集団発動 / データ整合性疑い
   - 手順: ① incident_id 発番 (INC-YYYY-MMDD-NNN) → ② Sentry / 構造化ログ / Webhook ログ / Unipile 履歴 を correlation_id で串刺し → ③ ステータスバナー & ステータスページ更新 (5 分ごと) → ④ 解消後 72h 以内に postmortem 公開 (`/notifications?type=incident` から到達可) → ⑤ 関連 SLO 違反のエラーバジェット集計を更新
   - SLA: 発番〜初報 10 分、postmortem 公開 72h。
