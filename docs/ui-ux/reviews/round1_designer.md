## 良い点 (3 つ)

1. **状態機械を IA・色・アイコン・並び順に貫通させる設計** (§1.1, §3.3, §21.2): 11 状態 × 色トークン × アイコン × 説明 × 遷移図が一気通貫。これにより一覧・タイムライン・通知すべてに同じ語彙が降りる。B2B SaaS で最も崩れやすい所を最初に押さえている。
2. **「AI is a Draft, not a Decision」の徹底** (§1.2, §6.4, §9.2): 全 AI 出力に「編集 / 採用しない / 根拠を見る」3 操作必須・ナレッジ参照ハイライト・送信 2 段階 (3 秒 Undo) など、HITL の一貫した語彙設計。LinkedIn 規約と日本語商習慣の両方に効く。
3. **安全側デフォルトの徹底**: ローンチ時は Draft 停止 (§5.1)、レビュー必須が初期値 (§9.1)、上限到達時は「停止」既定 (§6.8)、危険操作は二要素再認証 + Undo/復元 (§17, §1.2 Reversible First)。「BAN されない」をブランド軸にした主張と整合。

## HIGH (必修・Phase1 着手前にブロッカー)

### H1. ペルソナ × 画面マッピングが宣言止まり、JTBD と画面が結ばれていない

- 該当箇所: §2.1〜2.2, §4.1
- 問題: §4.1 の「主要ユーザー」列に P1〜P4 を当てるところまではあるが、(a) **JTBD J1〜J6 と画面の直接対応表がない**、(b) P4 (情シス／鈴木) の動線は §6.10 セキュリティと §6.11 データ削除のみで、**初回オンボーディングに P4 専用ゲート (SCIM / IP / 監査ログ初期設定) が存在しない**。Owner 初期セットアップで P4 が後追いする現状フローでは、Enterprise 商談で必ず詰む。
- 推奨改善:
  - §2.2 の直後に「**JTBD × 画面 × KPI** のクロス表」を追加し、各 JTBD に「主担当ペルソナ / 入口画面 / 完了判定イベント (PostHog)」を 1 行で書く。
  - §5.1 F-01 オンボーディングを「Owner ルート」と「Admin (情シス) 招待ルート」に分岐し、後者では **SSO / IP 制限 / 監査ログ保管期間 / リージョン選択** を 1 画面で選ばせる。
  - §6.10 セキュリティに「初期設定チェックリスト」(SSO / 2FA 強制 / 監査ログ通知 / リージョン) を常設し、未完了は黄色バッジで Owner ダッシュにも表示。
- 改善後の期待効果: Enterprise 評価における初回 PoC 通過率向上。P4 が Owner を経由せず単独で監査要件を完了でき、SSO 未設定での無防備運用を防げる。

### H2. HITL「レビュー必須 → 自動送信」昇格条件が UX として未定義

- 該当箇所: §9.1
- 問題: 「3 通連続の人間採用ログ + 同意ダイアログ」で自動送信に降りる、と書かれているが、(a) **どの粒度** (アカウント単位 / キャンペーン単位 / メッセージ種別単位) かが未定義、(b) **降格条件** (= 自動 → レビュー必須に戻る条件) が無い、(c) 「採用ログ」とは「AI Draft をそのまま送信」か「編集後送信」か曖昧。降格条件がないと安全モード発動以外で戻せず、品質劣化に気付けない。
- 推奨改善:
  - §9.1 にミニ状態機械を追加: `REVIEW_REQUIRED ⇄ AUTO_ELIGIBLE ⇄ AUTO_ON`、各遷移の条件・粒度・通知先を表で。
  - 昇格条件: 「**同一キャンペーン × 同一テンプレ** で 3 通連続、編集距離 ≤ 10%、返信率 ≥ ベースライン、過去 7 日 FAILED 0 件、Owner 同意」と定量化。
  - 降格条件: 「過去 24h で `safe_mode.triggered` / 規約違反疑い / 編集距離 > 30% が 3 通連続 / 返信率 50% 低下」のいずれかで自動 `REVIEW_REQUIRED` 復帰、Owner にメール。
  - 切替 UI を §6.5 / §6.10 のどこに置くかを明記（推奨: キャンペーン詳細 §6.2 の「設定」タブ + アカウントカード §6.5）。
- 改善後の期待効果: 自動送信の安全な昇格／降格が UI 上で可視化され、Operator が勝手に自動化して炎上する事故を構造的に防止できる。「下書く AI」原則 (§1.1) が運用フェーズでも保たれる。

### H3. 状態アイコンに絵文字 (🔍📄✅🚫⏳🤝✉️💬📅🏁⚠️) を採用、§14 の「絵文字過多 NG」と矛盾

- 該当箇所: §3.3, §6.5 (🟢🟡🔴), §13, §14
- 問題: (a) 絵文字はフォント / OS / プラットフォームで字形がバラつき、Windows / macOS / Linux / iOS で印象が変わる。Noto Sans JP と並ぶと **ベースラインずれ** が発生する (特に「🤝」「📅」)。(b) スクリーンリーダー読み上げが OS 依存 (NVDA で「ハンドシェイク絵文字」、VoiceOver で「Handshake」など読まれ、§13 の「意味を成す」要件と衝突)。(c) §14 で「絵文字過多 NG」と明示しているが本書自身がアイコンに絵文字を使い、原則と実装が矛盾。
- 推奨改善:
  - **アイコンは Lucide / Phosphor などの SVG アイコンセットに統一**。状態表は「アイコン名 (SVG) + ARIA ラベル + 表示テキスト」に書き換え。
  - 各状態の `aria-label` を 1 行で定義 (例: `<span aria-label="承認待ち">⏳</span>` ではなく `<HourglassIcon aria-label="承認待ち" />`)。
  - 絵文字は「サイドバー絵文字 (🏠🎯👥💬...)」を含め全て SVG アイコンに置換。§3.1 のサイドバー記法も修正。
  - §13 のチェック項目に「アイコンは画像/SVG であり、絵文字を意味伝達に用いない」を追加。
- 改善後の期待効果: WCAG 2.2 AA・スクリーンリーダー一貫性・印刷／PDF 出力・ダーク／ライト両対応で同じ意味を保てる。デザインシステム移植時の崩壊を防ぐ。日本語 B2B 商習慣（メールに絵文字を入れない文化）とも整合。

### H4. 受信箱「3 秒 Undo で送信」は B2B 取引の文脈で危険

- 該当箇所: §6.4 (S10 会話画面)
- 問題: 「クリック → 3 秒で送信されます／取消」は Gmail 風で個人向けには良いが、(a) LinkedIn DM は **送信後の取消不能** (Unipile 経由でも同じ)、(b) 機微情報 (見積額・契約名) を含む B2B メッセージで 3 秒のラグは「ボタンを押した瞬間に止まらない」体験となり、心理的安全性が低い。(c) 機微情報警告 (§6.4 黄色警告) と 3 秒 Undo の関係が未定義 (警告中も 3 秒 Undo に流れるのか?)。
- 推奨改善:
  - 送信フローを「**確定モーダル + Undo 併用**」に変更:
    - レビュー必須モード: 確定モーダル (送信先 / 文字数 / レート残量を再表示) → 即送信、送信後 5 秒だけ Undo (LinkedIn 側送信前のキューに留めるサーバ側遅延)。
    - 自動送信モード: ジョブキューで送信、送信前 60 秒間はダッシュから「キャンセル」可能 (§9.1 の昇格条件と連動)。
  - 機微情報警告がある場合は **3 秒 Undo を無効化し、必ず確定モーダル**。
  - §21.3 API 連携面に「Unipile 送信は内部キュー経由 (delay 5s)、UI の Undo はキューから DELETE で実現」と明記。
- 改善後の期待効果: 「ボタンを押した瞬間に止まらない」を解消し、B2B 取引の慎重さに合った操作感に。誤送信時の真の救済 (Undo) が機能する。

### H5. リード一覧の「保存ビュー」「タグ」「セグメント」が IA・コンポーネントカタログから抜けている

- 該当箇所: §3.2, §6.3, §8
- 問題: (a) §6.3 リード一覧のフィルタに `[+保存ビュー]` ボタンはあるが、保存ビューの編集／共有／既定化／削除画面・URL 設計が未定義。(b) 「タグ」が一括アクションに登場するが、タグ管理 (作成 / マージ / 削除 / 権限) の画面が無い。(c) §8 にビュー / タグ用コンポーネントの定義がない。SDR (P2) の日次運用で最も使う機能なのに IA 上は不可視。
- 推奨改善:
  - URL: `/leads?view=:viewId` でビュー切替、`/settings/leads/views` にビュー管理画面、`/settings/leads/tags` にタグ管理画面を追加。
  - §6.3 に「ビュータブ (固定: 全て / 自分の担当 / 要返信 / + ユーザー作成)」を追加し、固定ビューと保存ビューの違いを明示。
  - §8 に `SavedViewTabs`、`TagManager`、`TagChip` を追加し、タグの色 / 排他性 / 階層の有無を定義 (推奨: 排他なし / 階層なし / カラー 8 種固定)。
  - 権限マトリクス (§6.9) に「ビュー作成 / 共有」「タグ作成 / 削除」を追加。
- 改善後の期待効果: P2 (SDR 佐藤) の日次運用 (担当リードを瞬時に開く) が高速化。タグの統制が効き、3 ヶ月後にタグが乱立する典型的事故を防げる。

### H6. 受信箱の「会話オーナーシップ・ルーティング・SLA」が不在

- 該当箇所: §5.3, §6.4, §11
- 問題: 返信が来た時に「**誰の受信箱に入るか**」「**未対応 SLA**」「**他メンバーへの引継**」が未定義。(a) 1 アカウントを複数メンバーで共有運用する場合の挙動が不明。(b) §5.5 退職者ハンドオフでは「進行中スレッド → 別メンバーへ移管 or 自動応答停止」と書いているが、通常運用での移管 UI が無い。(c) 返信に対する SLA (例: 24h 以内) の可視化が無く、P1 (営業マネージャ) の管理視点が抜けている。
- 推奨改善:
  - §6.4 に「担当者バッジ」「@メンション」「移管ボタン」を追加し、Owner / Admin / Manager は他メンバーの受信箱を閲覧可能 (§6.9 権限と整合)。
  - §11.1 に「未対応 SLA 超過」レベル (Warning) を追加、超過スレッドはリスト先頭に固定。
  - §6.1 ダッシュ「注意が必要なもの」に「未対応 SLA 超過 N 件」を追加。
  - 受信箱フィルタに「自分の担当 / 未割当 / @メンション」を追加。
- 改善後の期待効果: P1 のマネジメント JTBD (J5 レポート、属人化解消) が実現。商談化機会の取りこぼし (リードが冷える) を SLA で構造的に防止。

### H7. ステータス バナー「閉じれない」と WCAG/ARIA の兼ね合いが未定義

- 該当箇所: §6.5 安全モード, §10.3, §12.4
- 問題: 「持続バナー (消えない)」「ステータスバナー (閉じれない)」とあるが、(a) スクリーンリーダーが画面遷移ごとに毎回読み上げる挙動の対処 (`aria-live` のレベル、`role="alert"` か `role="status"` か) が未定義。(b) 全画面に常駐すると上部の縦スペースが圧迫され、リード一覧の仮想化スクロール領域 (§6.3) と競合。(c) Critical なのに「閉じれない」と「Snooze 可能か」の区別がない (例: 安全モード解除待ちで 6h 作業する場合、ヘッダ常駐は集中阻害)。
- 推奨改善:
  - 4 段階に分ける: `inline (画面内)` / `sticky-top (上固定 / 折りたたみ可)` / `persistent (折りたたみ可だが閉じれない)` / `blocking (モーダル)`。安全モードは `persistent` + 折りたたみ時はアイコンのみ常駐。
  - ARIA: Critical = `role="alert" aria-live="assertive"` を初回のみ、以降は `role="status" aria-live="polite"`。同じ通知を再発火しない `dedupe key` を定義。
  - §13 のチェック項目に「アラート再発火制御」を追加。
- 改善後の期待効果: 視覚 / 聴覚両ユーザーの集中を阻害せず、安全モードの重要性は保てる。テーブルの高さ問題が解消し、§15 のスクロール 60fps を担保しやすい。

### H8. キャンペーン作成ウィザードに「離脱・再開・コラボ編集」フローが無い

- 該当箇所: §5.2, §6.2
- 問題: 5 ステップで多数の入力を要求するが、(a) Draft 自動保存の挙動 (頻度・コンフリクト解決) が §8 Stepper の説明 1 行のみ、(b) 別メンバーが同時に同じ Draft を編集した場合の挙動 (last-write-wins / ロック / マージ) が不在、(c) 「招待された Operator が途中まで作って Manager に承認を出す」レビュー導線が無い。チームで使う SaaS で頻発する「複数人で詰める」運用が抜けている。
- 推奨改善:
  - §6.2 にステップ毎の「自動保存タイムスタンプ + 編集者表示」を追加 (例: `2 分前に佐藤さんが保存`)。
  - 同時編集はロック方式 (Operator 編集中は他は閲覧のみ、5 分非操作で解放) を明記。
  - §5.2 に分岐: Operator が `Step5 → 承認待ち` でローンチを Manager に依頼するフロー (`status: PENDING_APPROVAL`)。承認画面はキャンペーン詳細にバナー表示。
  - 権限マトリクス (§6.9) に「キャンペーン承認」を追加。
- 改善後の期待効果: P3 (1 人代理店) は影響少だが、P1+P2 の協業フローが現実的に機能。Operator 単独でローンチ → 炎上を防ぐガバナンス層を獲得。

## MEDIUM (推奨・Phase1 中に対応)

### M1. KPI 数式の透明性は良いが、信頼区間 / サンプルサイズ警告がない

- 該当箇所: §6.1, §16.2
- 問題: 「承認率 = CONNECTED / PENDING」のような率は分母が小さい時に大きく振れる。サンプル N=12 で「承認率 50%」と表示されると意思決定を誤らせる。
- 推奨改善: KPI カードに `(N=12, 信頼区間広め)` の薄字・分母 < 30 の時は黄色バッジ「データ少」を出す。ホバーで Wilson 信頼区間。
- 改善後の期待効果: P1 (営業マネージャ) が初週の偽シグナルで判断する事故を防止、計測の透明性 (§16.2 の主張) が小サンプル時にも成り立つ。

### M2. オンボーディング ICP ヒアリング 3 問の中身が未定義

- 該当箇所: §5.1
- 問題: 「ICP ヒアリング (3 問)」とだけあり、何を聞くか・どう自動キャンペーン Draft に変換するかが空欄。Phase1 の中核体験。
- 推奨改善: 3 問を本書で定義 (推奨: 「①誰を狙う / ②なぜ買う / ③成功事例 1 件」)、各回答 → ICP フィルタ・メッセージテンプレ・ナレッジチャンクへの分解ルールを §5.1 に追記。
- 改善後の期待効果: 「10 分で最初の発見まで」(§5.1 成功条件) が再現可能になる。

### M3. キャンペーン詳細 (S06) のワイヤーフレームが無い

- 該当箇所: §3.2, §4.1, §6 (見当たらず)
- 問題: §3.2 で 4 タブ (概要 / リード / メッセージ / 設定) を宣言しているが、§6 にレイアウト・主要ウィジェット・空状態が無い。Phase1 の中核画面の一つ。
- 推奨改善: §6 に S06 のセクションを追加 (タブごとに KPI / リードテーブル / 配信スケジュール / A/B 結果 / レビューキュー / 設定編集ロック)。
- 改善後の期待効果: 実装着手前に空白を潰し、Phase1 のスコープが明確化。

### M4. 通知センターと「注意が必要なもの」 (§6.1) の役割分担が曖昧

- 該当箇所: §6.1, §11.3, §17
- 問題: ダッシュ「注意が必要なもの」(行動可能のみ)、通知センター (情報通知)、ヘッダ通知ベル の 3 箇所がある。同じイベントが 2〜3 箇所に出るルールが書かれておらず、ユーザーは「どれを見ればよいか」混乱する。
- 推奨改善:
  - 「**Action Required は 3 箇所すべて、Warning は通知センター + ベル、Info はベルのみ**」のマトリクスを §11.3 に追加。
  - ダッシュ「注意が必要なもの」の各カードを通知センターと同一 ID で結び、片方既読化 → もう片方も既読化を仕様化。
- 改善後の期待効果: 「通知の二重表示で何も解決しない SaaS」のアンチパターンを回避。

### M5. リード詳細のドロワー設計が浅い

- 該当箇所: §6.3
- 問題: 「右ペイン 60% でリード詳細 (モーダルではなくドロワー)」とのみ。タイムライン / シグナル / プロファイル / アクション / メモ の構造未定義。§4.1 で S08 単独画面の存在も宣言されており、ドロワー (60%) と単独 (100%) の使い分けが不明。
- 推奨改善: §6 に S08 のレイアウトを追加 (3 セクション: ヘッダ / タブ: タイムライン・プロファイル・メッセージ・メモ / アクション)。一覧からはドロワー、深堀り時にドロワー右上「拡大」ボタンで `/leads/:id` 単独画面へ。
- 改善後の期待効果: P2 (SDR) のリード深堀り体験が一貫し、URL 共有 (Slack で「このリード見て」) が成立する。

### M6. CRM 連携の双方向同期の競合解決ルールが未定義

- 該当箇所: §6.6
- 問題: 「同期方向: 双方向 / 一方向 (推奨)」とあるが、双方向時の競合解決 (last-write-wins / フィールド優先 / 手動マージ) が無い。
- 推奨改善: フィールド単位で「Source of Truth (本プロダクト / CRM / 最新)」を選べる UI、競合発生時のレビューキュー、`integration.conflict_detected` イベント発火を追加。
- 改善後の期待効果: HubSpot との運用で Owner 上書き事故が防げる。Enterprise 評価で必ず聞かれる項目。

### M7. ナレッジの責任表示・誤情報警告が不足

- 該当箇所: §6.7
- 問題: 「🔴 古い情報 / 矛盾検出」バッジは良いが、(a) 誰が何を根拠に「古い」と判定したか、(b) AI 出力に古いナレッジが使われた時の事後通知、(c) ナレッジ削除時に過去のメッセージ「根拠を見る」がリンク切れになる扱いが未定義。
- 推奨改善:
  - チャンク単位に「最終確認日 / 確認者」を表示、90 日経過で警告。
  - ナレッジ削除は「アーカイブ (引用先からは参照可能 / 新規利用不可)」を中間状態として設ける。
  - AI 出力の「根拠を見る」にナレッジ更新後の差分を表示。
- 改善後の期待効果: ナレッジが陳腐化しても監査・レビューが追跡可能。法務・コンプライアンスの懸念に応える。

### M8. モバイル仕様が宣言止まりで設計されていない

- 該当箇所: §3.1, §18
- 問題: ボトムタブ 5、FAB、受信箱・会話・通知・ダッシュ要約の MVP、と書かれているが、ワイヤー・タッチターゲット・片手操作・キーボード吸着の具体仕様が無い。SDR (P2) は外出先で返信対応する想定なら必須。
- 推奨改善: §18 に「会話画面モバイル」「受信箱モバイル」のワイヤーを 2 枚追加。AI Draft の折りたたみ高さ、送信ボタン位置 (右下 FAB 化)、SafeArea 対応、キーボード表示時の `dvh` 採用を明記。
- 改善後の期待効果: P2 の外出運用が成立し、MVP 評価フェーズで「モバイルが使えない」差し戻しを防げる。

### M9. エラーメッセージのテンプレ集が無い

- 該当箇所: §12.3, §14
- 問題: 「原因 + 直し方を 1 文で」原則は良いが、実装時の参照テンプレが無い。チームが書くと表現がバラつく (敬体 / 体言止め / 命令形が混在)。
- 推奨改善: 付録に「日本語エラーテンプレ集」(20 件目安) を追加。例: `400 入力エラー → 「<項目名>は<制約>です。<どう直すか>」` / `403 権限不足 → 「この操作は<必要ロール>のみ可能です。Owner にロール変更を依頼してください。」` / `429 レート → 「LinkedIn 側の制限により待機しています。あと約 N 分で再開します。」`
- 改善後の期待効果: エラー文言の一貫性が担保され、日本語 B2B トーンが Phase1 終了時点で統一される。

### M10. 監査ログ (§19) の検索・出力 UX が薄い

- 該当箇所: §3.2, §17
- 問題: 「監査ログ閲覧」が権限マトリクスにあるのに、画面仕様が §6 に存在しない (§3.2 にだけ URL `/audit`)。Enterprise 商談で頻出。
- 推奨改善: §6 に S19 のレイアウトを追加 (タイムライン / 検索: actor / target / action / from-IP / 日付、CSV 出力、SIEM webhook 設定)。
- 改善後の期待効果: P4 の Enterprise セキュリティ要件に応えられる。

### M11. ローカリゼーション英語版で破綻しやすい箇所が未対策

- 該当箇所: §14, §7.2
- 問題: 「商談化」「申請中」など日本語固有語が UI 全体に散在。i18n キー設計、英訳の合意 (例: 商談化 = "Meeting Booked" / "Opportunity Created"?)、文字長伸び (英語は約 1.3〜1.5 倍) によるテーブル列幅 / バッジ折返しの考慮が無い。
- 推奨改善: §14 に「i18n キー命名規則 + 英訳辞書 (20 状態 + 主要ボタン)」を追加。タイポグラフィ §7.2 に英語時の最大文字幅とトランケーションルール (`...` / 中央切り詰め) を明記。
- 改善後の期待効果: Phase2 英語版で UI が壊れない。

## LOW (任意・Phase2 以降)

### L1. ダッシュボードのカスタマイズ性
ウィジェット並び替え / 非表示 / 期間既定の保存。Phase3 のカスタムダッシュ前提なら Phase1 から「期間既定」だけは個人設定可能に。

### L2. 受信箱のキーボードショートカット ヘルプモーダル
`?` キーで一覧表示。§6.4 で `J/K/R/E/S` を定義しているのでヘルプ画面が欲しい。

### L3. KPI カードの単位表示の統一
「商談化数」は件数、「承認率」は%、「送信」は件数。万 / 億表記の発火閾値 (§14: 1 万以上で「1.2 万」) を §7.2 数値フォーマットに明記。

### L4. ファネルチャートの段階定義の可変性
ICP・キャンペーン目的により「QUALIFIED → MESSAGED」が無いケースがある。`COMPLETED` を直接目指す調査キャンペーン (J1 リサーチ) で 2 段階のファネルが必要。

### L5. 利用上の注意 (§6.12) の改版通知
規約改版時に再同意 UX (送信前ダイアログ強制) を Phase2 で。

### L6. AI ドラフト案 A/B/C の差分可視化
3 案がほぼ同じ時にユーザーが疲れる。差分行のハイライト (`+追加 / -削除`) を Phase2 で検討。

### L7. ヘルプ / 改善要望 (§3.1) の動線
サイドバーにあるが、フィードバック後の透明性 (受領 / 検討中 / リリース) が無い。Canny 風ロードマップを Phase3 で。

### L8. プラン超過時の従量課金 UX
§6.8 で切替可能とあるが UX 詳細無し。Phase2 で「超過アラート → 一時停止 → 従量同意」のフローを定義。
