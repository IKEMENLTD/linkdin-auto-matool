## NEW HIGH 指摘 (v1.1 改訂で発生 / 残存)

### H-N1. Critical 通知の「ON 強制」 vs 「Owner のみ OFF 可」が同一文書内で矛盾 (Round1 H-3 派生)
- 該当箇所: §11.1 (Critical: 既定 = "ON 強制") / §11.2.1 末尾 (「**Critical の OFF はロール: Owner のみ可能**。OFF にすると監査ログに残る」)
- 問題:
  - 「強制 ON」の文字通りの解釈は **Owner であっても OFF にできない** であり、§17 の Safe Defaults 思想とも整合する。
  - 一方 §11.2.1 末尾は **Owner なら OFF 可** と明記しており、ロックアウト時の救済措置 (§5.9 Break-Glass) と取り違えている可能性。
  - 監査ログに残せば OFF 可とすると、攻撃シナリオ「侵害された Owner 認証 → Critical OFF → CircuitBreaker / 安全モード発動を隠蔽 → 大量誤送信」が成立する。Round1 H-3 で警戒した「不可逆事故」の典型ケース。
- 推奨:
  1. **チャネル別** に強制度を変える: in-app は **完全強制 (Owner も OFF 不可)**、Email は冗長化前提で OFF 不可、Slack / SMS だけは Owner 設定可、と表で明示。
  2. もし Owner OFF を許容するなら、OFF 操作自体を Critical イベント化 + 共同 Owner にメール通知 + 24h 後に自動 ON 復帰。
  3. §11.1 の「ON 強制」をサブカラム化 (`UI 強制 / メール冗長化 / Slack 任意`) で表に書き直し、§11.2.1 と §27.4 (HITL 降格時の Critical) と合わせる。
- 期待効果: Critical の「最後の砦」を制度設計で守り、Round1 で挙げた「請求停止・BAN のような不可逆事故」を防ぐ。

## MEDIUM 指摘

### M-N1. §24.1 SLO 表の「エラーバジェット」列の値が SLO/期間と整合しない (NEW)
- 該当箇所: §24.1 SLO 一覧
- 問題:
  - 「Web (App) TTI P95 < 3.0s / エラーバジェット 5h / 30d」とあるが、TTI は **レイテンシ SLI** であり可用性 SLO ではない。レイテンシ SLO のエラーバジェットは「P95 を超えた測定窓の割合」で表すべき (例: "P95 < 3.0s を 99% の 5min 窓で達成")。`5h` は単位の取り違え。
  - SSE 接続維持率 99.5% / 30d → 正解は 30d × 24h × 0.5% = **3.6h**。表は `3.6h` とあり OK。だが「アラート: 切断率 > 0.5% / 5min」が **SLO と同じ閾値** で、エラーバジェット消費中盤での早期警戒 (例: 0.2% / 5min) が不在。違反確定後にしかアラートされない設計はランブックと噛み合わない (M-N6 と関連)。
  - Webhook P99 < 30s には期間ウィンドウとエラーバジェット列が空欄。
- 推奨:
  1. SLO 表に `SLI 種別 (latency / availability / freshness)` カラムを追加し、レイテンシ SLO は「目標 P95 を満たした測定窓 / 全測定窓 ≥ X%」の形式に統一。
  2. アラート閾値はバーンレート (例: 1h で 14.4 倍 / 6h で 6 倍 / 24h で 3 倍) に書き直し、「SLO 違反確定」と「早期警戒」を分離。
  3. Webhook P99 にも 30d バジェット (例: P99 違反 ≤ 1% 窓) を入れる。
- 期待効果: SLO ガバナンス文書として外部 (Enterprise / 監査) に出せる質になる。

### M-N2. §11.2.1 のエスカレーション SLA が 5 段階で過密、アラート疲労リスク (NEW)
- 該当箇所: §11.2.1 (T+0 / T+1 / T+5 / T+15 / T+30)
- 問題:
  - T+0 (バナー) / **T+1 (Email)** / T+5 (SMS / 電話) / T+15 (副管理者) / T+30 (CSM) は階段が細かすぎ、誤検知 1 回で Email / SMS / 電話・CSM が立て続けに鳴る。
  - 受信側の電話 / SMS のオプトイン取得・国際電話料金・コスト上限のガードが §11.2.1 にない。
  - SMS / 電話チャネルは PII (電話番号) の保管箇所・退職者ハンドオフ時の更新フロー (§5.8) と紐付いていない。
- 推奨:
  1. 階段を **3 段** に圧縮: T+0 (in-app) / T+5 (Email + Slack) / T+15 (SMS + 副管理者) / T+30 (CSM)。
  2. SMS / 電話は Enterprise オプションのみ、`/settings/team` の連絡先項目 (E.164 / 国コード必須) と監査ログを連携。
  3. 「同一 incident_id × 同一受信者は 24h で SMS / 電話 1 回まで」というレート上限を明文化。
- 期待効果: アラート疲労による無視・コスト暴走を抑止。

### M-N3. 監査ログから incident_id への動線が UI 仕様にない (Round1 M-5 残存)
- 該当箇所: §17 監査ログ / §24 (incident_id 発番) / S19
- 問題: §12.3.1 で incident_id を 4 識別子に分離した一方、`/audit?incident_id=...` のフィルタや system actor 行のリンクが UI 仕様に書かれない。
- 推奨: S19 に `actor_type = user | system | external_webhook` を必須カラム化、system 行は `incident_id` で `/status/{incident_id}` と Sentry transaction に直リンク。`/audit/exports` (S27) も同様に incident_id 付与。
- 期待効果: 監査・コンプライアンス対応の所要時間短縮 (Round1 M-5 と一致)。

### M-N4. 従量課金中の可視化が「持続バナー」から「KPI カード」に降格 (NEW)
- 該当箇所: §22.3「従量課金中は KPI カードに『従量課金中: 残 ¥X』を常時表示」
- 問題:
  - Round1 H-4 推奨では「ダッシュボード上部に持続バナー (注意色) で常時可視化」と要望していたが、v1.1 では KPI カード化されダッシュ S03 のみで可視化。**他画面 (受信箱 / キャンペーン詳細 / `/connections`) では見えない**。
  - 月末まで誰もダッシュを見ないテナントでは、従量課金 ON で進行していることを Operator が気付けない。
- 推奨: §11.2.2 のステータスバナーと同じレイヤで「課金状態バナー」を追加 (severity = `notice`、Critical ではないので閉じれる、ただし 24h で再表示)。
- 期待効果: H-4 で警戒した「ON 放置による月末事故」の二次防護。

### M-N5. 配信ステータスがメッセージバブルに反映されない (Round1 H-5 派生残存)
- 該当箇所: §6.4 S10 会話画面 / §21.3 API 連携面
- 問題: S25 `/jobs` で横断ビューは出来たが、Round1 H-5 推奨 #2 「各メッセージバブルに `送信中 / 送信済 / 配信確認済 / 配信失敗 (理由)` の 4 状態 + 自動バックオフ残時間」は依然未記述。ユーザーは S10 と S25 を行き来しなければ「自分の送信が届いたか」が分からない。
- 推奨: ConversationBubble (§8) に `delivery_status` プロパティ + アイコン (Lucide: `Clock` / `Check` / `CheckCheck` / `AlertCircle`) を必須化。失敗時のみ S25 への deep link。
- 期待効果: サポート問い合わせ削減 (Round1 H-5 期待値と同じ▲30-40%)。

### M-N6. SSE 早期警戒・再接続後の再同期プロトコルが §23.3 と §24.1 で不整合 (NEW)
- 該当箇所: §23.3 (`Last-Event-ID` で再送 / backoff 1-2-4-8-30 + jitter) / §24.1 (SSE 接続維持率 99.5%)
- 問題:
  - §23.3 は再接続プロトコルを定義するが、§24.1 のアラート (`切断率 > 0.5% / 5min`) が **SLO 確定値と同値** で、Round1 M-3 で要望した「再接続後の整合性回復」「10 秒で再接続できなければ read-only モード」が UI 仕様に降りていない。
  - SSE が黙って劣化した場合 (確立はするがイベントが届かない) を検知する死活監視が未定義。
- 推奨:
  1. SSE クライアントから 30s ごとの heartbeat イベント (`event: ping`) を仕様化、3 回連続欠落で UI を read-only + 黄色バナー。
  2. 再接続成功時に `since=<last_event_id>` で差分取得 → 完了で「最新化されました」トースト (Round1 M-3 推奨)。
  3. アラート閾値を `0.5%` (確定) と `0.2% / 1h` (早期警戒) の 2 段に分離。
- 期待効果: 「返信に気づかなかった」インシデントを抑止。

### M-N7. §17.4 の「降格モード」が §27 HITL ステートマシンに登場しない (NEW / 仕様間矛盾)
- 該当箇所: §17.4 「AI 自動応答は **降格モード** で対応」 / §27 (`REVIEW_REQUIRED` / `SEMI_AUTO` / `FULL_AUTO`)
- 問題: §17.4 の「降格モード」は §27 のいずれの状態にも対応せず、Indirect Prompt Injection 検知時の挙動が **どの状態に遷移するのか曖昧**。実装時に「現状維持で警告だけ」になり防御が骨抜きになるリスク。
- 推奨: §27.3 降格条件に「§17.4 Quarantine ヒット → `REVIEW_REQUIRED` に強制降格、Owner レビュー後に手動再昇格」と明記。RB-07 にも対応する手順を追加。
- 期待効果: LLM01:2025 (Indirect Prompt Injection) の防御を仕様レベルで担保。

## NEW 良い点 3 つ

1. **§12.3.1 で識別子を 4 階層に分離**: `incident_id (人間可読)` / `correlation_id (UUID v7)` / `trace_id (OTEL)` / `request_id (xreq-)` を提示先と用途で明確に分けたのは SRE 設計として模範的。コピーボタンのフォーマット指定 (`INC-XXXX / corr=...`) まで具体的。

3. **§5.9 Break-Glass + §26 脅威モデル**: Owner SSO ロックアウト復旧フローを「24h 一時昇格 + 監査ログ BREAK_GLASS フラグ + Owner メール通知」で仕様化。情シス (P4 鈴木) ペルソナの最大恐怖シナリオを正面から扱っており、Enterprise 商談で詰まらない。
