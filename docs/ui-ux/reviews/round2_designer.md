## Round1 HIGH 解消チェック

- **唯一の残課題**: §28 表に **JTBD 完了判定 PostHog イベント** が紐付いていない（Round1 推奨「JTBD × 画面 × KPI」のうち KPI 列が抜け）→ MEDIUM 級として降格、§16.1 のイベント表とのクロスリンクで対応可能

### H2. HITL 自動送信 昇格／降格条件 → ✅ **完全解消**

- §27 で `REVIEW_REQUIRED → SEMI_AUTO → FULL_AUTO` の 3 状態を厳密定義
- §27.2 昇格条件: 採用率 ≥ 80% / 失敗率 < 1% / DLP 0 / Owner 2FA。SEMI → FULL は 30 日連続稼働 + 4-eye principle（Owner + Admin 2 名）まで踏み込み、当初 Round1 で要求した粒度を超える
- §27.3 降格条件は §9.2.1 CircuitBreaker と参照統一、復旧条件まで明記
- §9.2.1 に 6 トリガー（失敗率 / 同一テンプレ連投 / レート / DLP / Indirect Prompt Injection / 警告）を表で網羅
- §27.4 でヘッダ常設バッジと進捗バーの UI まで定義
- 期待効果: Operator が勝手に自動化して炎上する事故が **構造的に発生不能** に。本書の最大の進化点

### H3. 絵文字依存と §14 自己矛盾 → ⚠️ **部分解消（HIGH 再掲）**

- 状態機械 §3.3 は **完全に Lucide SVG 化** され、aria-label を日本語固定で網羅。状態 chip は色 + アイコン + 文字の三重表現が原則として徹底（最大の前進）
- **しかし以下に絵文字が残存し、§14「絵文字過多 NG」と矛盾が継続**:
  - §3.1 サイドバー記法: `🏠 ダッシュボード / 🎯 キャンペーン / 👥 リード / 💬 受信箱 / 📚 ナレッジ / 🔌 接続 / ⚙️ 設定 / 🆘 / 🔔`（9 個）
  - §6.1 ダッシュ ワイヤー: `🏠 ダッシュボード / 🔔 注意が必要なもの / 🎯 直近キャンペーン`
  - §6.3 リード一覧: `🎯 全キャンペーン横断 / 🎯 まだリードがいません`
  - §6.5 LinkedIn アカウント管理: `🟢🟡🔴` 状態（§3.3 で SVG 化したのに同じドキュメント内で絵文字記法が残存）
  - §6.7 ナレッジ: `🟢🟡🔴` 品質バッジ
  - §12.2 空状態テンプレ: `🎯 まだキャンペーンがありません`
  - §1.1.1 NSM 表 / §11.1 Critical 通知の引用などにも残存
- **これは Round1 H3 の最重要指摘そのもの** —— アイコン辞書だけ SVG 化しても、ワイヤー記法・サイドバー・空状態に絵文字が残っていると実装者は両方を見て混乱し、結局絵文字で実装される。`PageIcon` `Bell` `Target` `Search` などの Lucide アイコン名で全置換するか、`[icon: Home]` のような表記規約を §3.1 に明記すべき
- **HIGH として再掲**: H3-v2 として後段に詳細

### H4. 3 秒 Undo の B2B リスク → ✅ **完全解消**

- §6.4 で「クリック → 確認モーダル（送信先・本文・参照ナレッジを再確認）→ 確定で 5 秒キュー Undo（実送信は 5s 待機、Esc / Cmd+Z 対応）」へ刷新
- 確認モーダルが **必ず** 入るため「ボタンを押した瞬間に止まらない」体験は解消
- §17.5 で機微情報 DLP がさらに強化され、価格 / 契約条件は **送信ブロック + Manager 承認**、個人情報列挙は **強制ブロック**。Round1 推奨を超える厳格化
- CHANGELOG に明記、設計意図のトレーサビリティも担保
- Round1 で残課題とした「Unipile キュー経由送信 (delay 5s) 内部実装」は §21.3 / §23.4 に同等の記述あり（自動送信は ✗、Optimistic 禁止）

### H5. リード一覧の保存ビュー / タグ / セグメント → ⚠️ **部分解消**

- §3.2.1 で「保存ビュー / セグメント / タグ」のセマンティクスを定義（ビュー = フィルタ条件 + 列構成 + ソート + 既定アクション、タグ = 複数付与可、セグメント = 動的タグ）
- §17.1 ABAC で `tag_scope` 属性が登場

  - **保存ビュー管理画面** (`/settings/leads/views`) の URL・共有 / 既定化 / 削除 UI が無い
  - §6.3 リード一覧のフィルタには `[+保存ビュー]` だけで、ビュータブ（固定: 全て / 自分の担当 / 要返信 + ユーザー作成）の UI が未追加
  - §8 コンポーネントカタログに `SavedViewTabs` `TagManager` `TagChip` が **無い**
  - §6.9 権限マトリクスに「ビュー作成 / 共有」「タグ作成 / 削除」の行が **無い**

### H6. 受信箱の会話オーナーシップ / SLA → ✅ **完全解消**

- §25 章まるごと新設、§25.1 オーナーシップ表（MESSAGED / REPLIED / MEETING / ESCALATED ごとに一次担当・共同担当・不在時代替）、§25.2 ルーティングルール UI（`/inbox/settings/routing`）、§25.3 SLA（一次応答 2h、超過 30 分前アラート、超過後赤フラグ）、§25.4 共有受信箱の権限（Operator は担当のみ表示、Manager 以上は全表示、機微会話は Owner / Admin のみ展開）
- §5.6.2 F-06c でルーティング判定フローが宣言、§5.8 退職者ハンドオフが Wizard 化
- 期待効果は完全に達成 —— P1 のマネジメント JTBD (J5) が成立

### H7. ステータスバナーと WCAG / ARIA → ✅ **解消**

- §11.2.2 で `role="alert" + aria-live="assertive"` の HTML スニペットを提示、`data-incident` 属性で重複抑止 (24h クールダウン、サーバ側 dismiss テーブル)
- 「閉じれない」のは Critical のみ、Warning は `aria-controls` で折りたたみ可、バナー高 96px max + 詳細リンクという仕様まで明記
- Round1 で要求した 4 段階分け（inline / sticky-top / persistent / blocking）の **完全な命名は無い** が、実質的に「sticky-top（折りたたみ可）」「persistent（Critical のみ閉じれない）」「blocking（モーダル）」が記述されているため OK
- 軽微な残課題: §10.3 安全モードバナーで「消えない」「持続バナー」と書かれた古い表現と、§11.2.2 の新仕様が両立しているか統合説明が欲しい → LOW として後段

### H8. ウィザード離脱・再開・コラボ編集 → ✅ **完全解消**

- §5.6.1 F-06b で「離脱・再開・コラボ・承認」を完全定義
  - **離脱**: 1.5s debounce で Draft 自動保存、離脱バナー + ディープリンク
  - **再開**: ダッシュ S03 の「続きから再開」カード
  - **コラボ**: 同時編集者アバター（Liveblocks/Yjs/Live のいずれか、§23 参照）
  - **承認**: Operator が「ローンチ申請」、Manager 以上にワンクリック承認導線
  - **差分**: ローンチ後の編集は再承認必要
- §6.9 権限マトリクスへの追加は文中宣言止まりで **表自体に「キャンペーン承認」行が無い** → LOW として後段
