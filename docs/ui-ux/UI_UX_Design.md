## 1. プロダクト原則 / デザイン原則

### 1.1 プロダクト原則

1. **送る前にレビューできる、を最優先**
   AI ドラフトを「送る AI」ではなく「下書く AI」と位置付ける。デフォルトはレビュー必須、慣れてから自動送信に降りる導線を用意する。
2. **アカウント保護がブランドの軸**
   どんな機能よりも「BAN されない」体験。レート閾値、ウォームアップ、警告サインは UI 上で常に可視化する。
3. **日本語 B2B の自然な距離感**
   英語圏 SaaS 直訳の語気・絵文字過多は採用しない。敬体ベース、句読点・改行は読みやすさ最優先。
4. **状態機械を UI のメンタルモデルにする**
   `DISCOVERED → ENRICHED → QUALIFIED → PENDING → CONNECTED → MESSAGED → REPLIED → MEETING/COMPLETED` を全画面で同じ語彙・色・順序で使う。

### 1.1.1 North Star Metric（NSM）と先行指標

| 種類 | 指標 | 定義 | 目標 |
| --- | --- | --- | --- |
| **NSM** | Reply Received per Active Account / 週 | 週 1 回以上送信したアカウント当たりの **新規返信件数** | β: 5 / Phase2: 8 / Phase3: 12 |
| Activation | A1 | サインアップ後 7 日以内に「初 REPLIED」到達 | 35% |
| Activation | A2 | サインアップ後 14 日以内に「初 MEETING」到達 | 18% |
| Retention | R1 | M1 → M2 継続率 | 85% |
| Retention | R2 | アカウント当たり月次アクティブ率 | 90% |
| Engagement | E1 | 週次ログイン日数中央値 | 3 日 |
| Health | H1 | 安全モード発動率（30 日） | < 2% |
| Health | H2 | AI ドラフト人手採用率（編集後送信を含む） | > 60% |

UI への露出ルール:
- ダッシュ S03 では NSM を **最も大きい数値**として表示し、A1/A2 進捗を Onboarding チェックリストで併記
- すべての KPI カードは「定義式」「データ更新時刻」「サンプル期間」を hover で開示（信頼性 UX）
- NSM の前期比は「±%」と「件数差分」の両方を表示し、向きは矢印 + 色 + 文字の三重表現

### 1.1.2 アクティベーション復帰導線（リテンション設計）

```
Day 0  サインアップ完了 → Welcome モーダル + オンボ Wizard
Day 1  「初 LinkedIn 接続が未完了」なら 09:00 にメール + アプリ内バナー
Day 3  「未送信のキャンペーン Draft」がある場合、再開ディープリンク付きメール
Day 7  「7 日以内 REPLIED 0」なら ICP 緩和提案 + 1on1 設定リンク
Day 14 「MEETING 0」なら CSM へ自動エスカレーション（対象 Owner のみ）
Day 30 「ログイン無し」なら休止プラン提案メール（解約防止）
```

休止プラン: 月額 **通常プラン額の 50%** / 送信機能停止 / 受信のみ継続 / データ 90 日保持。詳細は §22.1 を参照。チャーン耐性のキー機能。

### 1.2 デザイン原則（OKR ではなくレビュー基準）

| 原則 | 定義 | 検証方法 |
| --- | --- | --- |
| Clarity over Density | 情報密度より「次に何を見ればよいか」が明確 | 5 秒テスト：初見でユーザーが「次の操作」を口頭で言えるか |
| Safe Defaults | 危険な操作（自動送信、無制限スケール、共有）はデフォルト OFF | 設定画面の初期値レビュー |
| Reversible First | 送信・削除・連携解除は **24h Undo / 7d 復元** を優先 | 各破壊操作のロールバック仕様レビュー |
| AI is a Draft, not a Decision | AI の出力は常に編集可能 / 棄却可能 / 出典明示 | 全 AI 出力に「編集」「採用しない」「根拠を見る」3 つのアクションがある |
| Status is a First-class Citizen | 状態（Lead/Account/Job）は色だけでなくラベル・アイコン・テキストの三重表現 | カラーブラインド検証、スクリーンリーダー検証 |

## 2. ペルソナと JTBD

### 2.1 主要ペルソナ

#### P1. 田中 健司 — 営業マネージャ（30 名規模 SaaS の VP of Sales）
- 関心: 商談数の積み上げ、メンバーごとの活動量の見える化、アカウント停止リスク
- 痛み: SDR を雇うコスト、属人化、レポートを毎週手作業で作っている
- KPI: 月 80 商談 / SDR 4 名分の代替

#### P2. 佐藤 美咲 — SDR / IS 担当（実働 1〜3 アカウント運用）
- 関心: 1 日に何件送ればよいか、返信が来たら誰がいつ何を返すか
- 痛み: 自分の LinkedIn が止められないか不安、どのリードが「アツい」か分からない
- KPI: 自分の月コネクト承認率、返信率、商談設定数

#### P3. 山本 拓也 — 起業家 / 1 人代理店
- 関心: 安く・早く・自分 1 人で回したい、複数案件を並行
- 痛み: ツール乱立、結局スプレッドシート、CRM とつながらない
- KPI: 顧客獲得コスト、月 5 案件並行運用

#### P4. 鈴木 由香 — 組織管理者 / IT・情シス
- 関心: SSO・監査ログ・データ保管リージョン・退職者のアクセス遮断
- 痛み: シャドー IT、データ持ち出し、GDPR / 個人情報保護法対応
- KPI: 監査対応工数、インシデント 0 件

### 2.2 Jobs-To-Be-Done

| # | Job | When | I want to | So I can |
| --- | --- | --- | --- | --- |
| J1 | ICP 定義 | 新製品をローンチした時 | 製品とターゲット会社像を入力するだけで検索クエリ・メッセージ雛形を得たい | 1 日で初回キャンペーンを動かせる |
| J2 | 安全に大量に送る | 月初 | アカウントごとに上限と人間っぽい間隔で配信したい | LinkedIn に怒られない |
| J3 | 返信に即応 | 営業時間内に返信が来た | 文脈を AI が要約し、3 案返信を提示してほしい | 取りこぼしと長文労力を削れる |
| J4 | 商談化を逃さない | 返信に温度がある | 即 CRM に連携し、カレンダー打診まで自動でやりたい | リードを冷やさない |
| J5 | レポートを自動で | 週次定例の前 | チーム / アカウント / キャンペーン別に成果を 1 画面で見せたい | 報告作業を 0 にする |
| J6 | アカウントを増減 | チーム拡大 / 退職 | 安全にアカウントを足し / 切り、引き継ぎたい | 個人依存を消す |

#### 3.2.1 保存ビュー / セグメント / タグ（リードと受信箱の横断機能）

すべての一覧画面（`/leads` `/inbox` `/campaigns/:id?tab=leads`）で同じセマンティクスの「ビュー」を持つ:

```
[フィルタ] → [+保存ビュー] → ワークスペース共有 or 個人
ビュー = フィルタ条件 + 列構成 + ソート + 既定アクション
タグ   = リード/会話に複数付与可（例: "ABM-Tier1", "競合検討中"）
セグメント = 動的タグ（条件で自動加入、例: "VPoE × Series B"）
```

ビュー/タグ/セグメントは権限スコープに従い表示制御。共有ビューは Owner/Admin/Manager のみ作成可。

### 3.3 メンタルモデル: 状態機械を IA に揃える

すべての一覧・タイムライン・通知の **並び順・色・アイコン** は次の表に統一する。アイコンは `lucide-react` の **線画 SVG** を使用し、絵文字には依存しない（B2B トーン保持・i18n 安全・スクリーンリーダー安定）。

| 状態 | 表示ラベル(ja) | ラベル(en) | 色トークン | Lucideアイコン | aria-label | 説明 |
| --- | --- | --- | --- | --- | --- | --- |
| DISCOVERED | 発見 | Discovered | `--state-neutral` | `Search` | 「発見済」 | 検索でヒットしたが未調査 |
| ENRICHED | 調査済 | Enriched | `--state-info` | `FileText` | 「調査済」 | プロフィール詳細取得済み |
| QUALIFIED | 適合 | Qualified | `--state-positive-soft` | `CheckCircle2` | 「適合」 | AI 判定で ICP 合致 |
| DISQUALIFIED | 除外 | Disqualified | `--state-muted` | `MinusCircle` | 「除外」 | 適合しない（理由表示） |
| PENDING | 申請中 | Pending | `--state-warning-soft` | `Hourglass` | 「申請中」 | コネクト送信済み・承認待ち |
| CONNECTED | 接続 | Connected | `--state-info-strong` | `Link2` | 「接続済み」 | 1st コネクション |
| MESSAGED | 送信済 | Messaged | `--state-progress` | `Send` | 「送信済」 | 初回メッセージ送信済 |
| REPLIED | 返信あり | Replied | `--state-positive` | `MessageSquare` | 「返信あり」 | 返信受信（要レビュー） |
| MEETING | 商談化 | Meeting | `--state-success` | `CalendarCheck` | 「商談設定済」 | カレンダー設定済 |
| COMPLETED | 完了 | Completed | `--state-success-strong` | `Trophy` | 「完了」 | CRM 同期完了 |
| FAILED | 失敗 | Failed | `--state-danger` | `AlertTriangle` | 「失敗」 | 送信失敗・規約違反疑い |
| EXPIRED | 期限切れ | Expired | `--state-muted` | `Clock` | 「期限切れ」 | コネクト申請が承認されず期限到達 |
| SAFE_MODE | 安全モード | Safe Mode | `--state-danger` | `ShieldAlert` | 「安全モード作動中」 | 自動停止中 |
| QUARANTINED | 隔離 | Quarantined | `--state-warning` | `BadgeAlert` | 「隔離中」 | DLP/規約検査で保留 |

実装ルール:
- 状態 chip は **アイコン + 色 + 文字** の三重表現を必須（カラーブラインドおよび SR 対応）
- `aria-label` は表の通り日本語固定。ラベル(en) はクラス名/イベント名/CSV出力に使う
- 並び順は遷移順（DISCOVERED → ... → COMPLETED、例外として FAILED/EXPIRED/SAFE_MODE/QUARANTINED は末尾）
- 色トークンの数値は §7.1 を参照、ダーク／ライトで自動切替

## 5. 主要ユーザーフロー

### 5.1 F-01 初回オンボーディング（〜10 分で「最初の発見」まで）

```
Sign up → メール認証 → ワークスペース作成 → 業種・人数選択
  → LinkedIn 接続(Unipile OAuth) → ICP ヒアリング(3 問) 
  → 自動でキャンペーン Draft 生成 → 「最初の 10 リード」プレビュー
  → 送信モード選択(レビュー必須/自動)→ ダッシュボード
```

### 5.11 F-11 プラン上限 / 従量課金切替（保護機構）

```

```

```

│ 受信箱   │ 山田 太郎 — Acme Inc / VPoE           │ プロフィール  │
```

- キーボード: `J/K` で前後スレッド、`R` 返信、`E` アーカイブ、`S` スヌーズ、`G then I` 受信箱へ

- AI Draft の `生成中` は「タイプライターアニメ」を **使わない**（操作の遅延に見えるため）。代わりに左上ローダーと `ESCで中止`

## 9. HITL（Human-in-the-Loop）UX

### 9.1 デフォルトはレビュー必須

- 新規キャンペーンは「レビュー必須」が初期値
- 「自動送信」に切り替えるには 3 通連続の人間採用ログ + 同意ダイアログ
- 自動送信中でも **危険語** が含まれる文面はキューで止める

### 9.2 AI 出力の必須要件

すべての AI 出力に次の 3 操作が並ぶ:

```
[編集して送信]   [この案は使わない]   [根拠を見る]
```

「根拠を見る」を押すと、参照したナレッジチャンクと、相手プロフィールの該当箇所をハイライト。

### 9.2.1 自動送信 CircuitBreaker（必須）

自動送信モードが ON でも、以下のいずれかで **即座にレビュー必須に降格**:

| トリガー | 降格条件 | 復旧条件 |
| --- | --- | --- |
| 失敗率 | 直近 24h で送信失敗率 > 5% | 失敗率 < 1% を 24h 維持 |
| 同一テンプレ連投 | 同じ AI ドラフトが 50 回以上連続採用 | テンプレ更新（差分 ≥ 30%） |
| 単位時間レート | 直近 1h で送信が平常時平均 × 3 倍 | 1h 平常値復帰 |
| DLP ヒット | 送信前検査で機微情報検出が 1 件以上 | 設定見直し + 24h クールダウン |
| Indirect Prompt Injection 疑い | 受信メッセージに指示文パターン検出 | Owner レビュー後に再開 |
| 警告イベント | LinkedIn からの警告サイン 1 回以上 | Owner レビュー + 48h クールダウン |

CircuitBreaker 発動時は:
- 全画面に持続バナー（§11.3 参照、Owner ロール以外は閉じれない）
- 監査ログに `CIRCUIT_BREAKER` イベント
- Slack/Email に Critical 通知

### 9.3 失敗時のリカバリ

- AI が止まったら「人間ドラフト」テンプレに即フォールバック
- 「AI が下書きできませんでした」を **責任ある言葉** で表示する（「申し訳ありません、もう一度お試しください」ではなく原因と回避策を書く）

## 10. レート制御 & ウォームアップ UX

### 10.1 共通バー

`RateLimiterMeter` を全画面に常駐させる候補（受信箱/会話/送信ボタン上）。

```
[アカウント: 林 翔太]   本日 18/25 送信  クールダウン 12 分
```

### 10.2 ウォームアップ計画の説明

- 14 日間、日次上限を 1/3 → 2/3 → 100% に段階アップ
- 段階アップは **失敗 0 件 + 返信率 > 5%** が条件
- 条件未達なら自動で停滞、UI で次の解除条件を表示

### 10.3 安全モード自動切替

- 連続失敗 5 件 / 24h 警告 ≥ 1 / Unipile 4xx 急増 → そのアカウント全送信を即停止
- ダッシュ・受信箱・会話画面に持続バナー（消えない）
- 解除フローは F-06 参照、解除には Owner 承認

## 11. 通知 / リアルタイム / ステータス設計

### 11.1 通知レベル

| レベル | 例 | チャネル | 既定 |
| --- | --- | --- | --- |
| Info | 日次サマリ | アプリ内 / メール（週次） | OFF |
| Action Required | 返信あり / 要レビュー | アプリ内 / プッシュ | ON |
| Warning | ウォームアップ停滞 | アプリ内 / メール | ON |
| Critical | 安全モード発動 / 認証期限切れ | アプリ内（強制） / メール（強制） / Slack（任意 ON） | チャネル別強制度 (下表) |

#### 11.1.1 Critical 通知 チャネル別 強制度

| チャネル | Owner | Admin | Manager | Operator | Viewer |
| --- | --- | --- | --- | --- | --- |
| アプリ内バナー | **OFF 不可** | **OFF 不可** | **OFF 不可** | **OFF 不可** | **OFF 不可** |
| メール | OFF 可（Owner本人のみ／監査ログ記録） | OFF 不可 | OFF 不可 | OFF 不可 | OFF 不可 |
| SMS / 電話 | 任意 ON | 任意 ON | — | — | — |
| Slack / Teams 連携 | 任意 ON | 任意 ON | 任意 ON | — | — |

「アプリ内バナー」は **誰も OFF にできない**（持続バナー §11.2.2）。Owner はメール通知だけ自分宛にOFFにできるが、監査ログに `CRITICAL_EMAIL_DISABLED_BY_OWNER` を残し、副 Owner / Admin にメール通知される。

### 11.2 リアルタイム

- 受信箱と会話画面は SSE（または WebSocket）で即時反映
- 切断時はバナー「リアルタイム接続が切れています — 再試行中」+ 自動再接続
- Optimistic UI を使うのは「下書き保存」「既読化」のみ。送信は Optimistic 禁止

### 11.2.1 Critical 通知のエスカレーション（必須 SLA）

```
T0       検知（自動 / 監視）
T+0:00   アプリ内 Critical バナー（持続、Owner 以外は閉じれない）
T+0:01   Email → Owner / 請求担当 / 緊急連絡先
T+0:05   未確認なら SMS / 電話（オプション、Enterprise）
T+0:15   未確認なら共同 Owner / 副管理者へエスカレート
T+0:30   未確認なら CSM が連絡
```

Critical 種別:
- 安全モード発動（アカウント停止）
- プラン上限到達（自動停止）
- 自動送信 CircuitBreaker 発動
- SSO / OAuth 認証期限切れ（72h 前から黄色、24h 前から赤）
- インシデント Sev1（サービス全停止）

ユーザー側設定:
- `/settings/profile/notifications` で個別 ON/OFF。**チャネル別の強制度は §11.1.1 の表に従う**（アプリ内バナーは全ロール OFF 不可、メールのみ Owner 本人 OFF 可）。OFF にすると監査ログに残り、副 Owner / Admin にメール通知

### 11.2.2 ステータスバナーの仕様（ARIA / 再発火 / 抑止）

```html
<div role="alert" aria-live="assertive" data-severity="critical" data-incident="INC-2026-0142">
  …
</div>
```

- `role="alert" + aria-live="assertive"` で SR に即時告知
- 同一 incident の重複表示を抑止（24h クールダウン、`localStorage` ではなくサーバ側 dismiss テーブルで管理）
- 「閉じれない」のは Critical のみ。Warning は `aria-controls` で折りたためる
- バナー高は最大 96px。それ以上は省略 + 「詳細」リンク
- バナー内のリンクは Tab 順序を画面の最前面に挿入、フォーカス可視

### 11.2.3 サービスステータスページ `/status`（S24）

- **Source of Truth**: バックエンドの `incidents` テーブル + `services` テーブル（`api`, `ingest`, `unipile-bridge`, `llm-anthropic`, `llm-openai`, `webhooks`, `web`）
- 各サービス: 4 段階（`operational` / `degraded` / `partial_outage` / `major_outage`）
- 公開フィード: JSON `/status.json` と RSS `/status.rss`
- 過去 90 日のアップタイム棒グラフ
- インシデント詳細: `incident_id` / 開始時刻 / 検知契機 / 影響サービス / 暫定対応 / 解消時刻 / Postmortem URL
- 顧客通知: Subscribe 機能（Email / Slack Webhook）
- 関連 Runbook: §24 参照

### 11.3 通知センター `S17`

- フィルタ: All / Action / Warning / Critical

### 12.1 ローディング順序


### 12.2 空状態テンプレ（S22）

すべて 4 要素: イラスト / 1 行説明 / 1 CTA / ヘルプリンク。例:

```
🎯 まだキャンペーンがありません
  最初のキャンペーンを 5 ステップで作れます
  [キャンペーンを作成]    [作り方ガイドを見る]
```

### 12.3 エラー（S23）

- 4xx は **何が悪いか + 直し方** を 1 文で（「入力 X が長すぎます。300 文字以内に収めてください」）
- 5xx は「サーバ側で問題が発生しています。インシデント番号 INC-XXXX を控えてサポートへ」と **incident_id** を必ず提示
- 401 はサインイン画面に戻すが、戻り先 URL を保持

#### 12.3.1 incident_id / correlation_id / trace_id の提示ルール

| 識別子 | 形式 | 提示先 | 用途 |
| --- | --- | --- | --- |
| `incident_id` | `INC-YYYY-NNNN`（連番） | エラーモーダル / ステータスバナー / メール | サポート連絡用、人間が読みやすい |
| `correlation_id` | UUID v7 | エラー詳細「コピー」アイコン | クライアント〜API〜キュー〜LLM 間の横串 |
| `trace_id` | OpenTelemetry trace ID | サポート向けの隠しフィールド | バックエンド観測性 |
| `request_id` | xreq- prefix | ヘッダ `X-Request-Id` | 個別リクエスト |

UI 表示:
- すべての 5xx / 4xx ≥ 500 の応答時間遅延では `incident_id` と `correlation_id` を表示
- 「コピー」ボタンで両方をクリップボードに（フォーマット: `INC-XXXX / corr=...`）
- サポート問い合わせフォームには自動的にこれらが事前充填
- ロールバック先: 直前のスクリーンショット ID（PostHog セッションリプレイ）も Owner / Admin には表示

- ステータスバナー: アプリ最上部、**閉じれない**（critical のみ）
- ステータスページへの導線

## 17. セキュリティ UX / 権限 / 監査

- セッション絶対有効期限: 8h（管理者画面 1h、Owner SSO 強制 4h）
- **Idle Timeout（無操作タイムアウト）**: 一般ユーザー 30 分 / Manager 以上 15 分 / Owner 10 分 / `/recovery/break-glass` 5 分。タイムアウト 1 分前に警告モーダル「あと 60 秒で自動ログアウトします」、操作で延長
- **デバイス管理**: `/settings/security/devices` で過去 90 日のログイン端末一覧（OS / IP / 国 / 最終アクセス）。「他のデバイスをすべてサインアウト」操作可、監査ログに記録
- 危険操作: 二要素再認証（送信先 CSV 出力 / プラン変更 / メンバー削除 / 自動送信 ON / Break-Glass / API キー再生成 / リージョン変更）
- 監査ログ: WHO / WHEN / WHAT / FROM-IP / FROM-UA / DIFF / `purpose`（理由メモ任意） / `correlation_id`
- **監査ログ append-only / 改竄耐性**:
  - 物理: WORM ストレージ（S3 Object Lock Compliance Mode、保持期間 SLO に従う）
  - 論理: 各エントリに `prev_hash` を持つ hash chain、毎日 root を Postgres と外部 KMS 両方に署名保存
  - UI: `/audit` には削除/編集ボタン無し（Owner でも不可）。誤投稿の訂正は **打ち消しエントリ追記**でのみ可能、`/audit` 画面で `signed_at` と「整合性検証 ✓ 2026-05-09 09:00 JST」を常時表示
  - 13ヶ月 (Enterprise) / 90 日（標準）の保持後に **アーカイブのみ削除可**（Owner 2 名 4-eye + 監査委員会承認）
- 機微情報マスク: 電話・メールはデフォルト `***`、表示は権限要
- API キー: 一度しか表示しない、再生成で前のキーは即無効、UI で警告
- 退会: 30 日 grace 後物理削除、確認に `DELETE` 文字入力必須

### 17.1 ABAC（Attribute-Based Access Control）

ロール（RBAC）だけでは不足。次の属性で重ね掛け:

| 属性 | 例 | UI |
| --- | --- | --- |
| `linkedin_account_owner` | リード／会話の所有 LinkedIn アカウントの担当者のみ参照可 | リード一覧で他担当行が `[非表示]` |
| `campaign_member` | キャンペーン参加メンバーのみ参照可 | プロジェクト型ワークスペース |
| `tag_scope` | タグ「ABM-Tier1」を持つリードは Manager 以上 | フィルタが自動で当たる |
| `region` | 顧客が EU 顧客の場合 EU リージョンのみ | 越境表示前に同意ダイアログ |
| `ip_range` | 社内 IP のみ閲覧可（`/settings/security`） | 拒否は明示理由 |

UI: 行が ABAC で見えない場合は「N 件は権限により非表示」と件数だけ示す（情報漏洩防止）。

### 17.2 CSV エクスポート（透かし / 上限 / 目的記録）

- エクスポート前にダイアログ:
  - **目的の記録**（必須、自由記述、監査ログに残る）
  - **対象範囲の確認**（件数 + プレビュー）
  - **2FA 再認証**
- 透かし: 各行に hidden 列 `__watermark` = `{user_id}|{export_id}|{epoch}`、メタファイルに同 ID
- 上限: 1 回 1 万行 / 1 日 5 万行 / Owner 解除可
- 出力後 7 日で URL 失効（署名付き）
- すべての DL は `/audit/exports`（S27）で誰がいつ何を出したか可視化、Owner は撤回可

### 17.3 PII サニタイズ（LLM プロバイダ送信前）

- 入力（プロフィール本文・受信メッセージ）から電話・メール・住所・マイナンバー・クレジット番号を **正規表現 + ML 検知** でマスク
- マスクは `[REDACTED:phone#1]` のようなトークンに置き換え
- LLM 出力に同トークンが残っている場合は元の値に復元（ローカルで保持したマップを使用）
- UI: AI ドラフトの「根拠を見る」モーダルで「マスクされた項目: 3 件」と件数表示
- ログ・トレースには **マスク後の値のみ** を残す
- ユーザーが元データの参照を必要とする場合は権限要 + 監査ログ記録

### 17.4 Indirect Prompt Injection 防御 UX

- 受信した DM 中の指示文（"Ignore previous instructions" 系のパターン、長文の制御プロンプト、リンクされた外部 URL）は AI に渡す前に **Quarantine（隔離）** に振り分け
- 隔離されたメッセージは S29 で人間レビュー、AI 自動応答は §27 HITL ステートマシンの **`REVIEW_REQUIRED` へ降格**（`SEMI_AUTO` / `FULL_AUTO` 双方からの強制降格）。再昇格には Owner 承認 + クールダウン
- AI ドラフトの根拠表示で「この返信文には指示文として解釈される可能性のある記述が含まれます」を明示
- 万一の誤動作のため、AI 自動応答は **常に編集可能 + 5s Undo + 24h 取消**

### 17.5 機微情報送信ブロック（DLP / 黄色警告では足りない）

| パターン | 旧（黄色警告のみ） | 新（v1.1） |
| --- | --- | --- |
| 電話番号 / メールアドレス | 警告 → 送信可 | 警告 → 送信は権限要（Operator 不可、Manager 以上） |
| 顧客固有用語 / 内部コードネーム | 警告 → 送信可 | 警告 → 確認モーダルで意図確認 |
| 価格 / 契約条件 / 値引き | なし | **送信ブロック** + Manager に承認依頼 |
| 競合社名 | なし | 警告 + 採用前に再確認 |
| 個人情報の本格列挙 | なし | **強制ブロック** |

### 17.6 DSAR / 削除波及スコープ

GDPR Art.17 / 個情法 28条 対応。`/settings/data` で削除リクエスト時に対象を可視化:

```
削除対象スコープ

☑ pgvector / 埋め込み              即時（再計算不可）
☑ S3 / 添付ファイル               即時（バックアップは 30 日後）
```

### 17.7 外部連携の同意撤回 UX

## 21. 付録: 用語 / 状態機械 / API 連携面

### 21.1 用語

### 21.2 状態遷移図（要点）

```
DISCOVERED → ENRICHED → (QUALIFIED | DISQUALIFIED)
QUALIFIED → PENDING → (CONNECTED | EXPIRED → DISQUALIFIED)
CONNECTED → MESSAGED → (REPLIED | NO_REPLY → cooldown)
REPLIED → MEETING → COMPLETED
任意 → FAILED（送信失敗・規約違反疑い）
```

UI ではこの遷移を **2 次元の進度バー** と **タイムライン** で表す。

### 21.3 API 連携面（UI 観点）

- Unipile OAuth: 別タブでポップアップ → 完了で親に postMessage、失敗時は明確なエラー
- Webhook 受信: 受信箱バッジを 5 秒以内に更新、未届のリトライ UI を保持
- 失敗系 4xx: 受信箱の該当メッセージに「再試行」「除外」「サポートに送る」の 3 ボタン
- レート 429: 自動バックオフ、UI には「混み合っているため待機中」と表示

### 23.2 キャッシュ命名規約（重複防止）

```
Next.js fetch cache key:        cache:<resource>:<id>:<params hash>
Next.js revalidateTag:          tag:<resource>:<id>     例: tag:campaign:42
TanStack Query key:             ['<resource>', id, paramsObj]
SSE channel:                    sse:<workspace_id>:<topic>:<entity_id?>
SWR fallback:                   _swr:<resource>:<id>
```

- Next の Server Cache と TanStack の Client Cache は **同じ resource のときだけ tag invalidation を共有**
- mutate 時は Server Action で `revalidateTag` + クライアントは `invalidateQueries` を 1 セットで呼ぶ helper を必ず使う
- グローバル `staleTime` は禁止、画面別に明示（`staleTime: 0` `30s` `5min` `Infinity` の 4 種から選ぶ）

### 23.3 リアルタイム配信基盤（SSE 採用）

- **既定: SSE（HTTP/2）**、フォールバックでロングポーリング
- 接続上限（HTTP/2）: 1 origin 100 接続 → 同時 1 ユーザー 1 接続 + workspace 跨ぎはチャンネル多重化
- メッセージ ID 必須（`Last-Event-ID` で再接続後の再送）
- backoff: 1s → 2 → 4 → 8 → 30 + jitter ±20%
- 切断検知 5s、UI バナー出現 3s 後（チラつき防止）
- 認可: workspace_id をクエリではなく Cookie 認証ヘッダで送る（CSRF 安全）
- スケール: Redis pub/sub で 5 万同時接続まで、超過時は Centrifugo / Ably に切替

### 23.4 Optimistic UI ポリシー（粒度を細かく）

| 操作 | Optimistic | 理由 |
| --- | --- | --- |
| 既読化 | ✓ | 失敗してもユーザー影響小 |
| Draft 保存 | ✓ | 失敗時はリトライ |
| タグ付与 / 外し | ✓ | 失敗時は rollback トースト |
| スヌーズ | ✓ | 失敗時 rollback |
| 送信（DM / コネクト） | ✗ | 二重送信リスク、必ずサーバ確定後に表示 |
| 状態遷移（QUALIFY 等） | ✗ | サーバ側の整合性が真 |
| 削除 | ✗ | 5 秒 Undo モデル |
| プラン切替 | ✗ | 課金影響 |
| メンバー権限変更 | ✗ | セキュリティ |

他セッションの SSE 受信と Optimistic の競合は **サーバ ID 一致なら無視 / 不一致ならサーバ優先で merge**。

### 24.2 Runbook（最低 5 件 + 拡張）

#### RB-01 安全モード発動オペレーション
1. 検知（自動）→ Banner 出現 + Owner にメール（< 60s）
2. SRE 当番が 5 分以内に初報（Slack #incidents）
3. 影響範囲の確認（該当アカウント、件数）
4. Unipile / LinkedIn 警告ログを確認
5. 30 分以内に推定原因を Postmortem ドラフトに
6. 解除条件達成まで送信停止維持

#### RB-02 Unipile 障害 / OAuth 大量期限切れ
1. ステータスフィードで `partial_outage` に切替（≤ 2 min）
2. 全テナントへ Critical 通知
3. 影響アカウントを `degraded` 状態に、UI は黄色バナー
4. 自動再認証メール送信（OAuth 期限切れ）
5. 復旧後 24h は手動の段階的再開

#### RB-03 LLM ベンダ障害フェイルオーバー
1. Primary（Anthropic）失敗率 > 5% を検知
2. 3 分以内に Secondary（OpenAI）へ自動切替
3. 切替時は AI ドラフトに「予備モデルを使用中」の小さなバッジ
4. KPI（生成品質）を 24h モニタ、低下時は手動復帰

#### RB-04 プラン上限 / 従量課金暴走の止血
1. 月次上限金額の 80% で警告
2. 100% で自動 OFF（1 分以内）
3. Owner / 請求担当へ通知
4. 過剰請求の場合は手動精算プロセスへ

#### RB-05 incident_id 発番 / Postmortem 公開
1. SLO 違反 5 分以上で自動発番（`INC-YYYY-NNNN`）
2. 初報 10 分以内
3. 解消後 72h 以内に Postmortem 公開（`/status` から）
4. アクションアイテムは Linear に自動起票

#### RB-06 退職者ハンドオフ緊急停止（追加）
1. HR 連携でアカウント無効化トリガ
2. すぐに該当ユーザーの Send 系全停止
3. ハンドオフ Wizard を強制起動
4. 24h 以内にハンドオフ完了しなければ Owner にエスカレ

#### RB-07 Indirect Prompt Injection 検出（追加）
1. DLP / プロンプトガードが検出
2. 該当スレッドを Quarantine（S29）
3. AI 自動応答を降格（CircuitBreaker §9.2.1）
4. Owner レビュー後に再開

- 一次応答 SLA: 受信から **2 時間以内**（営業時間 9-18 平日）
- SLA 超過 30 分前にアラート
- SLA 超過後は受信箱の該当スレッドが赤フラグ + チームスタンドアップに表示

### 25.4 共有受信箱の権限と表示

- Operator は **担当のみ表示**
- Manager 以上は全スレッド表示可
- 機微会話（DLP マーク）は Owner / Admin のみ展開可
- 受信箱の検索・エクスポートも ABAC を適用

## 27. HITL 昇格／降格 ステートマシン

### 27.1 状態定義

| 状態 | 説明 | 送信前レビュー | デフォルト |
| --- | --- | --- | --- |
| `REVIEW_REQUIRED` | 全送信が人間レビュー必須 | 必須 | 新キャンペーンの初期値 |
| `SEMI_AUTO` | 人間が監督、信頼スコア閾値を超えるドラフトのみ自動送信 | 一部 | 昇格条件達成時 |
| `FULL_AUTO` | 全送信が自動、CircuitBreaker のみで停止 | 不要 | 上級プラン + 解放後 |

### 27.2 昇格条件（厳密）

```
REVIEW_REQUIRED → SEMI_AUTO
  - 過去 30 日で人間採用率 ≥ 80%（採用 = 編集なし or 編集 < 20% で送信）
  - 過去 30 日の失敗率 < 1%
  - DLP ヒット 0 件
  - Owner が同意ダイアログを承認 + 2FA 再認証

SEMI_AUTO → FULL_AUTO
  - SEMI_AUTO で 30 日連続稼働
  - 平均返信率 > 5%
  - 警告イベント 0 件
  - Owner + Admin 2 名の承認（4-eye principle）
  - 月次審査必須
```

### 27.3 降格条件（自動）

§9.2.1 CircuitBreaker の発動条件と同じ。発動後の再昇格は **クールダウン後 + Owner 承認**。

### 27.4 UI 表示

- すべての画面ヘッダに `HITL: REVIEW_REQUIRED` バッジ（クリックで現在の昇格進捗を表示）
- ダッシュ S03 のオンボチェックリストに「自動送信を解放する条件」を進捗バーで
- 降格時は赤バナーと監査ログ + Slack/Email Critical

## 29. 文言ガイドライン / エラー文テンプレ集

### 29.1 トーン原則

- 敬体（です / ます）固定
- 主語を省きすぎない（「保存しました」より「変更を保存しました」）
- ユーザーの責任を断定しない（「あなたの操作が誤っています」NG）
- 行動可能（次に何をすればよいかを必ず書く）
- 技術用語は最小限。必要なら括弧で補足
- 句読点「、。」、半角コロン `:` は英数列挙のみ

### 29.2 NG / OK 例

| シーン | NG | OK |
| --- | --- | --- |
| 検証エラー | `入力に誤りがあります` | `「業界」を 1 つ以上選んでください` |
| 認証期限 | `期限切れです` | `LinkedIn の認証期限が切れています。30 秒で再接続できます。` |
| 5xx | `エラーが発生しました` | `処理中に問題が発生しました（INC-2026-0142）。15 秒後に再試行できます。` |
| AI 失敗 | `AI が応答できませんでした` | `AI ドラフトを生成できませんでした。テンプレを使うか、もう一度試してください。` |
| 危険操作確認 | `本当によろしいですか？` | `この操作は 24 時間以内なら取り消せます。実行しますか？` |
| 空状態 | `データがありません` | `まだリードがいません。最初のキャンペーンを 5 分で作れます。` |

### 29.3 エラー文テンプレ集（コピペ可）

```
[400] 入力エラー:    「{field}」を{rule}にしてください
[401] 期限切れ:      セッションが切れました。サインインし直してください
[403] 権限不足:      この操作は{role}以上の権限が必要です
[404] 未検出:        該当の{resource}は見つかりません
[409] 競合:          他のメンバーが同時に編集中です。再読み込みして続けてください
[429] レート制限:    混み合っています。{seconds}秒後に再試行できます
[500] 内部:          処理中に問題が発生しました（{incident_id}）。サポートへ
[503] メンテ:        メンテナンス中です。{eta}に再開予定
[Network] 切断:      接続が切れました。再接続を試みています…
```

### 29.4 i18n 辞書ルール

- すべての文言は `messages/ja.json` `messages/en.json` に配置
- ICU MessageFormat を使う（複数形 / 性別 / 日付）
- プレースホルダは `{name}` 形式、HTML を含めない
- スクリーンリーダー専用は `srOnly` キーで明示

```json
{
  "inbox.unreadCount": "{count, plural, =0 {未読なし} other {未読 #件}}",
  "campaign.estReach": "推定到達 {count, number, ::group} 件"
}
```
